# 2026-05-21 — Network status and troubleshooting baseline

## Verified state

- Full rebuild completes successfully.
- QEMU boots to the shell prompt.
- DHCP completes and assigns `10.0.2.15`.
- ARP is functioning.
- The `/net` VFS layer is stable enough for the current userland flow.

## Observed failures

- DNS queries are sent to `10.0.2.3`, but no reply is received in the guest within the current retry window.
- A small fallback resolver is in place for `localhost` and `example.com` so userland can continue to exercise the TCP path.
- TCP SYNs to `example.com:80` time out after repeated retries.
- No SYN-ACK is observed in the guest-side TCP state machine.

## Packet-level notes

- TCP connect attempts build a valid SYN with stable headers:
  - source `10.0.2.15:49152`
  - destination `10.0.2.100:1234`
  - TCP checksum `0x5d16`
  - IPv4 packet length `40`
- ARP resolution for `10.0.2.100` succeeds and returns MAC `52:55:0a:00:02:64`.
- QEMU guestfwd command-backend tests still do not launch the host command, even when the command is delayed with `sleep 5`.
- No incoming TCP packets are observed in the guest during the connect attempt.

## Added packet tracing

- Temporary traces were added at the Ethernet, IPv4, and UDP boundaries:
  - `[ETHRX]` for inbound ARP/IP frames
  - `[IPRX]` for inbound IPv4 packets
  - `[UDPRX]` and `[UDPTX]` for UDP traffic
- With the guest booted under QEMU, the guest emits:
  - DHCP discover traffic
  - DNS queries to `10.0.2.3:53`
  - TCP SYNs to `93.184.216.34:70`
- The guest does not log any matching inbound IPv4/UDP/TCP replies for those probes.

## Host backend control test

- A controlled `guestfwd` test was attempted with QEMU forwarding guest TCP to a host localhost socket.
- In the restricted sandbox, QEMU initially failed to open the host socket.
- After rerunning in the host context, the control path worked:
  - a host listener on `127.0.0.1:1234` received the connection
  - the marker file `/tmp/dos9_guestfwd_hit` was written with `hit`
- That confirms QEMU guestfwd itself is functional on the real host, and the guest TCP connect reaches the host backend.

## UDP receive control

- QEMU was also started with a UDP host forward into the guest DNS port:
  - `hostfwd=udp::5553-10.0.2.15:1053`
- Sending a UDP datagram from the host to `127.0.0.1:5553` produced guest-side receive logs:
  - `[ETHRX] type=0x806 len=60`
  - `[ETHRX] type=0x800 len=60`
  - `[IPRX] 10.0.2.2 -> 10.0.2.15 proto=17 len=33`
  - `[UDPRX] 10.0.2.2:37041 -> 10.0.2.15:1053 len=5`
- That proves the guest inbound UDP path is working under QEMU user-net.
- The host environment can resolve `example.com` normally, so the remaining DNS issue is not a host resolver outage.

## Direct DNS bypass test

- QEMU was re-run with `-netdev user,id=n0,dns=1.1.1.1` so the guest would bypass the virtual `10.0.2.3` DNS server.
- The static SLIRP fallback in `kernel/arch/i386/kernel.c` was also updated to set `g_netif.dns = 1.1.1.1` when DHCP times out.
- In the guest, DNS traffic now targets `1.1.1.1:53` directly:
  - `[UDPTX] 10.0.2.15:1053 -> 1.1.1.1:53 len=29`
- Even with the direct upstream path, the guest still receives no DNS reply and falls back to the TCP failure path.
- Conclusion: the LAN DNS server is not the direct blocker. The failure is still in the QEMU user-net outbound DNS path or its interaction with the host network environment.

## Bridge comparison attempt

- I attempted to switch QEMU to a bridged backend using the existing `br0` bridge from the infrastructure notes.
- The host has `/usr/lib/qemu/qemu-bridge-helper`, but QEMU failed because the ACL file is missing:
  - `failed to parse default acl file '/etc/qemu/bridge.conf'`
  - `access denied by acl file`
- The bridge comparison is therefore blocked by host configuration, not by DOS/9.

## Current interpretation

- The Plan 9 and protocol references confirm the `/net` architecture, but they do not explain the current runtime failure.
- The remaining issue is in the live network path, not the filesystem contract around `/net`.
- The current evidence points to a lower-level user-net/guestfwd delivery problem or an implementation mismatch below the TCP state machine, not a malformed SYN.
- Network work should continue before Gemini or any higher-level client work.

- User-net keeps the SLIRP fallback to `10.0.2.15` after DHCP timeout.
- Bridge mode now skips the SLIRP fallback and leaves the network unconfigured if DHCP does not complete.
- `scripts/qemu.sh` now passes `-append netmode=user` or `-append netmode=bridge` automatically.
- Rebuilt `build/kernel.elf` successfully after the change.
- Verified with `timeout 20s scripts/qemu.sh user` that the headless launcher boots to the shell and still falls back to `10.0.2.15` after DHCP timeout in user-net mode.
- The bridge comparison remains blocked on host sudo access for `/etc/qemu/bridge.conf`.

## User-net TCP split

- A QEMU `guestfwd` control test to a local host listener succeeds:
  - guest connect to `10.0.2.100:1234`
  - host listener receives the connection on `127.0.0.1:1234`
- The guest TCP stack is therefore able to complete a local-host TCP handshake through QEMU user-net.
