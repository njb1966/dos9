What if DOS evolved in a different direction?

DOS/9 is not trying to recreate MS-DOS. It asks a different question: what would a small, understandable, shell-centered operating system look like if it borrowed the right ideas from Plan 9 while remaining a native kernel?

The shell is the OS

The command environment is not an afterthought. It is the primary interface, workflow, and control surface.

Everything is a file

Devices, processes, modules, and disks are exposed through a filesystem-oriented model.

Understandable by design

The goal is not maximal abstraction. The goal is a system a determined programmer can read, build, and reason about.

Current status

DOS/9 is already booting, scheduling, entering user mode, launching a shell, and exposing core system services through files and syscalls.

Built around real OS pieces

DOS/9 is a real kernel project, not a mock shell. The current system includes boot, memory, interrupts, scheduling, filesystems, syscalls, and user processes.

Kernel foundation

  • GRUB / Multiboot boot path
  • GDT, IDT, and PIC remapping
  • PS/2 keyboard and PIT timer
  • Paging and heap allocator

Process model

  • Preemptive scheduler
  • Ring-3 user execution
  • int 0x80 system calls
  • ELF loading

Filesystem-first design

  • VFS mount points
  • /dev keyboard and console
  • /proc process inspection
  • /disk DOS9FS storage

Shell

  • History and tab completion
  • Variables
  • if, for, loop, break
  • help, exec, run, rm, pid

Pipes and file descriptors

  • Anonymous pipes
  • dup and dup2
  • Pipeline execution
  • waitpid process control

Userland

  • Small libc
  • hello
  • cat
  • Shell launched from /disk/sh

Architecture

DOS/9 targets 32-bit x86 and favors directness over cleverness. It is designed for people who want to see how the machine fits together.

Native target

The system runs as a native kernel, currently developed and tested through QEMU using a cross-compiler workflow.

C with assembly where needed

The implementation favors plain C and small pieces of assembly for hardware-facing code, bootstrapping, and architecture boundaries.

Text mode matters

DOS/9 treats text not as a limitation, but as a durable interface for power users, scripting, debugging, and small-system clarity.

Hardware-friendly goals

The long-term aim includes booting on modest hardware, with “bootable on a Pentium” treated as a real design target.

Roadmap

These items are planned, not currently complete. DOS/9 is being built in visible stages.

TUI toolkit

A native text interface layer for future DOS/9 applications.

File manager and editor

Practical tools for living inside the system rather than only testing it.

Package system

A small, understandable way to install and manage DOS/9 software.

Networking

Networking support as a foundation for small-web tools and remote workflows.

Gemini, Gopher, RSS

Native tools for human-scale publishing and reading protocols.

DOSBox-style game support

Possible compatibility path for classic software without making DOS/9 a DOS clone.
